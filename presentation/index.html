The through-line
Every decision was about reducing the places where the model could fail — through grounding, validation, and human oversight — while being honest about where the gaps still exist and what a more robust production version would look like.

What this is

01 / overview

A web app that turns any topic into a 5-question multiple-choice quiz.

  • User submits a topic + difficulty
  • Server grounds the request in Wikipedia before generation
  • Claude generates questions via structured tool use (no free-form JSON)
  • Scoring is a pure function — never touches the LLM
  • Flagged questions hit a real endpoint, in the shape a review queue would consume

Built as an MVP — designed as production

Every shortcut is named explicitly, with the production swap called out in the same shape across the README. The pattern is the artifact.

Decision 1 — Split the models

04 / decisions
Match tier to task

Haiku 4.5 classifier · Sonnet 4.6 generator

Pre-flight classification runs on every submit — it's small, structured, fast. Question generation runs once per accepted topic — quality matters and is amortized.

Result: low cost/latency floor on rejection traffic, real quality where it ships.

Tradeoff: two model versions to track in the Anthropic console. Worth it for the clarity of separation.

Production extension

Ambiguous or high-stakes topics could escalate to Opus 4.7 with a longer-form chain-of-reasoning prompt. The split is already in place — escalation is a tier flag on the route.

Haiku 4.5 Sonnet 4.6 → Opus 4.7

Decision 2 — Defense in depth as mechanism

05 / decisions
Generator gets two tools, picks one

tool_choice: {type: "any"}

The generator can call generate_quiz or refuse_topic — Sonnet picks.

The classifier upstream is one layer. The system prompt is another. The two-tool choice is the third — implemented as an actual mechanism the model can invoke, not just a clause in a system prompt asking it to refuse.

Why this matters: a single forced tool would make "Layer 3" prose-only theatre. The model has to be able to refuse, structurally. 
// api/generate.ts (abridged)
const res = await anthropic.messages.create({
  model: GENERATOR_MODEL,
  tools: [generateQuizTool, refuseTopicTool],
  tool_choice: { type: "any" },
  system: [{ type: "text", text: SYSTEM,
    cache_control: { type: "ephemeral" } }],
  messages: [/* topic + wiki summary */],
});

const block = res.content.find(b => b.type === "tool_use");
if (block?.name === "refuse_topic") return refusal(block.input);
return quiz(block.input); // Zod-validated next

Decision 3 — Fail closed on missing grounding

06 / decisions
Hallucination prevention is the feature

No Wikipedia → no quiz

  • Server fetches Wikipedia REST before the Sonnet call
  • 404? Hit opensearch for a fuzzy match — handles "Eifel Tower" → "Eiffel Tower"
  • Still nothing? Reject the topic. No ungrounded generation. Ever.
Tradeoff: we reject some legitimate niche topics with no Wikipedia article. A production version falls back to Britannica or a curated topic library — never to ungrounded generation.

Why server-side?

Wikipedia content is user-influenced via topic choice. Fetching server-side gives one place to truncate, sanitize, and clamp the context window — the same place that injects it into the prompt. That's the prompt-injection defense seam.

User-visible proof

The SSE stage: 'wikipedia' phase shows "Reading Wikipedia…" in the UI. Grounding isn't hidden — it's the loading state.

Decision 4 — Typed retry by failure mode

07 / decisions

Tool use removes JSON-parse failure as a category. Three failure modes remain — each gets a different response.

FailureWhy it happensRetry
Transient API error429, 5xx, timeout1 retry, exp. backoff
Tool output shape OK, Zod failse.g. correct: "E", wrong count, dup options1 retry, validation error fed back
Model used no toolrefusal-style behaviourno retry — surface honestly
SDK retries disabled (maxRetries: 0) — stacking blind SDK retries on top of typed app-level retries hides timing from the user and the logs. One layer that I own, that I can reason about.

Decision 5 — AI does the writing, code does the math

08 / decisions
Where the LLM does and doesn't work

Scoring never touches Claude

Answer comparison, score arithmetic, per-question pass/fail — deterministic. Same input, same output. Testable without a stub.

The LLM writes questions and explanations. Everything around it is plain code.

Why this matters: reviewers can ask "what happens if the LLM is wrong?" The answer is "the wrong answer is still scored correctly against the answer key the LLM committed to." 
// src/composables/useScoring.ts
export function score(
  questions: GeneratedQuestion[],
  answers: Record<string, "A"|"B"|"C"|"D">,
): Score {
  const total = questions.length;
  const correct = questions.reduce((n, q) =>
    n + (answers[q.id] === q.correct ? 1 : 0), 0);
  return { correct, total };
}
// → 100% unit-tested · no Anthropic mock

Decision 6 — Three independent safety layers

09 / decisions
LAYER 1

Client-side validation

Length floor & ceiling. Submit disabled until input passes. No API call wasted on empty strings.

LAYER 2

Pre-flight classifier

Haiku checks viable · appropriate · intent. Distinguishes "not quizzable" from "careless language" from "deliberate intent".

LAYER 3

Generator refusal tool

Even after the gate passes, Sonnet can call refuse_topic with a category (harmful / ungroundable / ambiguous).

No single layer is relied upon alone. Each is independently testable — the classifier has a snapshot test on its prompt, the generator's tool schema rejects malformed output at the API, the client validator runs before any network call.

Decision 7 — Cache the static, pay for the dynamic

10 / decisions
Prompt caching as architecture, not config

Generator system block is cached

  • System prompt + difficulty rubric + tool schema → cache_control: { type: "ephemeral" }
  • Wikipedia summary + topic → uncached, per-request
  • Classifier prompts are too short to clear cache minimum — skipped
~90%
input token discount on cache-hit prefix · ~10× cheaper repeat generations

Production extension: per-topic Wikipedia summaries with a longer TTL when the same topic recurs.

Public AI endpoints are unpriced DDOS vectors

11 / cost

Public Vercel URL + an Anthropic API key = anyone can burn credits in a for loop. Three belts:

Belt 1 · Per-IP rate limit

Upstash Redis sliding window, ~10 generations/hour per IP. Stateless-serverless-safe (in-memory wouldn't work). Applied to /api/classify, /api/generate, /api/flags.

Belt 2 · Anthropic spend cap

Hard ceiling at the provider. If both my code and Upstash fail, the bill still can't run away.

Belt 3 · Cloudflare Turnstile

Named in the README, not built. Next layer for bot-class abuse beyond what IP rate-limiting catches.

Rate limit is the cost ceiling. Turnstile alone doesn't stop a determined script — IP rate-limiting does.

Bugs found · how I caught them

12 / bugs
BugSymptomFix
Internal error string leaks to UI Haiku refuses extreme content → fallback returns reason: "Classifier returned malformed input." which HomeView rendered raw to the user. Reason strings that look like internal errors get swapped for a generic safe message. HomeView never exposes raw result.reason.
Flag POST silently dropped from history Flagging from HistoryDetailView wrote to localStorage but never hit /api/flags — review queue invisible to silent ops. Composable now POSTs from both ResultsView and HistoryDetailView. User confirmation toast on both paths.
Direct POST bypassed classifier Adversarial test: POST /api/generate {"topic":"Methamphetamine"} skipped Haiku and produced a full quiz. Classifier now runs server-side inside /api/generate, not only on the client. Gate is structural, not client-trust.
Blocked topic emitted no telemetry Refusals weren't being captured anywhere — review queue saw nothing of the upstream gate. New /api/topic-block endpoint mirrors the flag-endpoint contract for blocked attempts.
Flag schema accepted negative timestamps POST /api/flags {"generatedAt": -1} returned 200. Added .min(0) to the Zod schema. One line; would have caught a class of corrupt-data bugs.

Testing strategy

15 / quality

Test the deterministic layer

  • Scoring — pure function, full unit coverage
  • Zod schemas — happy and adversarial inputs
  • Prompt construction — snapshot tests lock the prompt as a versioned artifact
  • Topic validation — length, character class, empty state
  • Storage envelope — version field + shape drift behavior
Don't test the LLM. It's non-deterministic, slow, and expensive. Treated as a tested-by-contract dependency — Zod validates at the boundary.

The flag button is the seed of HITL

17 / hitl

Structured signal · review-queue shape

Flag appears on the results screen — after the user sees the correct answer + explanation. That's when "wait, that's wrong" becomes meaningful.

Modal asks for category — inaccurate · ambiguous · poorly_worded · other. Free text doesn't get filled in; categories produce data a review queue can consume.

The endpoint is real. localStorage-only flagging is vapor. Storage backend swap is one config change. 
// POST /api/flags

  "quizId":      "qz_8a1f...",

  "topic":       "The Krebs Cycle",
  "generatedAt": 1747512030
}

Same Upstash rate limit applies. Same Zod boundary. Same logging contract.

What I'd build next

18 / next

1 · Streaming render

Parse tool-use input deltas, render each question as it lands. Drops perceived latency from ~6s to <1s. Biggest perceived-quality lift available.

2 · Per-topic Wikipedia cache

Same-topic re-generations near-instant. Extends the prompt-caching pattern from system block to per-topic context with longer TTL.

3 · Stubbed-Anthropic E2E harness

Covers happy path + each retry tier without paying per run. Lets every PR exercise the full pipeline deterministically.

Pattern: each next-step extends a seam that's already in the codebase — caching, streaming, validation — rather than introducing a new abstraction.

Through-lines · the things to remember

19 / summary
  1. Matched model tier to task. Haiku for the cheap pre-flight, Sonnet for grounded generation, Opus named for escalation.
  2. Defense in depth is mechanism, not prose. Two-tool choice makes Layer 3 real.
  3. Failed closed on missing grounding. No Wikipedia = no quiz. Hallucination prevention is the feature.
  4. Typed failure modes. Semantic vs. transient vs. refusal get different responses. SDK retries off — one layer that I own.
  5. Repeated MVP/production pattern. Five places. Local capture, named swap, same contract.
  6. Deterministic seams around the LLM. Scoring, validation, prompt construction are pure and tested.
  7. Cached the static, paid for the dynamic. ~90% input discount on the system block.
  8. Flag button as HITL seed. Structured signal in review-queue shape.
